Keeto is a module for OpenSSH that enables profile-based administration of access permissions in a central LDAP aware Directory Service, adds support for X.509 certificates and takes care of the distribution of key material in an automated and secure manner.

The fundamental idea behind is to leverage existing infrastructure usually found in larger enterprises for managing access to OpenSSH servers comprising the assignment and revocation of access permissions as well as the management of key material. In order to gain full advantage of Keeto a PKI, a Directory Service and a strong protection mechanism for private keys such as Smartcards should be present.

Having all access permissions in a central repository highly simplifies its management with regards to the assignment/revocation and overview of permissions. Keeto represents access permissions through different profile types that are assigned to OpenSSH servers centrally in a LDAP aware Directory Service such as OpenLDAP or Active Directory. Currently two different profiles are supported enabling either direct access or the access of an account on behalf of another.

Proper key management is vital to maintain the security of services. This includes the secure generation of key material with approved tools and security parameters, the secure distribution and installation of keys and an appropriate key lifecycle management including the revocation, regeneration and removal of keys.
OpenSSH uses its own key format that barely consists of the data needed for the cryptographic operation. In comparison to X.509 certificates there is

  • no binding to an identity
  • no integrity protection
  • no expiration
  • no mechanism for revocation
This properties can lead to either manual processes or insecure handling of the key material. In a worst case scenario OpenSSH keys are
  • generated with unknown and unapproved tools possibly using unsecure algorithms
  • transferred via non-integrity protected channels e.g. email or ticket
  • not verified over a secure channel before installation
  • deployed in an unsecure manner creating hard to detect backdoors
  • not regenerated in certain intervals
  • not removed from the server when access should not be permitted anymore
